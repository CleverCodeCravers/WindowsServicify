# R00009: Command-Injection-Schwachstellen beheben

## User Story

**Als** Administrator, der WindowsServicify auf einem Server einsetzt,
**moechte ich** dass ServiceName, DisplayName, Description und filePath aus der config.json validiert werden,
**damit** ein boeswillig manipuliertes config.json keine beliebigen Kommandos auf meinem System ausfuehren kann.

## Mehrwert

Die Konfigurationswerte werden direkt an `sc.exe` und PowerShell-Kommandos uebergeben. Ohne Validierung kann ein Angreifer, der Schreibzugriff auf die config.json erlangt, beliebige Systembefehle ausfuehren. Durch eine zentrale Whitelist-Validierung beim Laden der Konfiguration wird diese Angriffsflaeche vollstaendig geschlossen:
- ServiceName, DisplayName und Description werden gegen ein sicheres Zeichenmuster geprueft
- filePath wird auf Existenz und gegen Path-Traversal-Sequenzen validiert
- Ungueltige Werte fuehren zu einer klaren, informativen Fehlermeldung
- Die Validierung erfolgt zentral, sodass alle Konsumenten (LegacyInstallHelper, PowerShellInstallHelper, BackgroundService) automatisch geschuetzt sind

## Akzeptanzkriterien und Szenarien

### Szenario 1: Gueltige Konfiguration wird akzeptiert

**Given** eine config.json mit ServiceName "MyService", DisplayName "My Service", Description "A test service"
**When** die Konfiguration geladen wird
**Then** wird die ServiceConfiguration ohne Fehler zurueckgegeben

### Szenario 2: ServiceName mit Sonderzeichen wird abgelehnt

**Given** eine config.json mit ServiceName "MyService; rm -rf /"
**When** die Konfiguration geladen wird
**Then** wird eine ServiceConfigurationValidationException geworfen
**And** die Fehlermeldung benennt das ungueltige Feld "ServiceName" und den ungueltigen Wert

### Szenario 3: DisplayName mit PowerShell-Injection wird abgelehnt

**Given** eine config.json mit DisplayName "Test$(calc.exe)"
**When** die Konfiguration geladen wird
**Then** wird eine ServiceConfigurationValidationException geworfen
**And** die Fehlermeldung benennt das ungueltige Feld "DisplayName"

### Szenario 4: Description mit Backtick-Injection wird abgelehnt

**Given** eine config.json mit Description "Desc`whoami`"
**When** die Konfiguration geladen wird
**Then** wird eine ServiceConfigurationValidationException geworfen
**And** die Fehlermeldung benennt das ungueltige Feld "Description"

### Szenario 5: ServiceName mit Pipe-Operator wird abgelehnt

**Given** eine config.json mit ServiceName "Svc|net user hacker pass /add"
**When** die Konfiguration geladen wird
**Then** wird eine ServiceConfigurationValidationException geworfen

### Szenario 6: Leerer ServiceName wird abgelehnt

**Given** eine config.json mit einem leeren ServiceName ""
**When** die Konfiguration geladen wird
**Then** wird eine ServiceConfigurationValidationException geworfen
**And** die Fehlermeldung weist darauf hin dass ServiceName nicht leer sein darf

### Szenario 7: ServiceName mit erlaubten Zeichen (Bindestrich, Unterstrich, Punkt, Leerzeichen) wird akzeptiert

**Given** eine config.json mit ServiceName "My_Service-Name 2.0"
**When** die Konfiguration geladen wird
**Then** wird die Konfiguration ohne Fehler akzeptiert

### Szenario 8: Command mit Path-Traversal wird abgelehnt

**Given** eine config.json mit Command "..\..\..\..\Windows\System32\cmd.exe"
**When** die Konfiguration geladen wird
**Then** wird eine ServiceConfigurationValidationException geworfen
**And** die Fehlermeldung benennt das ungueltige Feld "Command"

### Szenario 9: Mehrere ungueltige Felder werden alle gemeldet

**Given** eine config.json mit ServiceName "Svc;bad" und Description "Desc$(evil)"
**When** die Konfiguration geladen wird
**Then** wird eine ServiceConfigurationValidationException geworfen
**And** die Fehlermeldung enthaelt beide ungueltigen Felder
