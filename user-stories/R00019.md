# User Story: R00019 - .editorconfig hinzufuegen

## Wer nutzt es?

Als **Entwickler im Team** moechte ich eine `.editorconfig`-Datei im Solution-Verzeichnis haben,
damit alle Teammitglieder unabhaengig von ihrer IDE oder ihrem Editor einheitliche Formatierungs-
und Coding-Style-Vorgaben verwenden.

## Mehrwert

Ohne `.editorconfig` haengt die Code-Formatierung von den individuellen Editor-Einstellungen ab.
Das fuehrt zu inkonsistenten Einrueckungen, Zeilenenden und Namenskonventionen. Code Reviews
werden durch Formatierungsunterschiede erschwert, und Git-Diffs enthalten unnoetige Aenderungen.
Eine `.editorconfig` legt diese Regeln projektuebergreifend fest und wird von allen gaengigen
IDEs (Visual Studio, VS Code, Rider, etc.) automatisch erkannt und angewendet.

## Szenarien

### Szenario 1: .editorconfig existiert im Solution-Verzeichnis

**Given** das Repository wurde geklont
**When** ich das Solution-Verzeichnis `Source/WindowsServicify/` oeffne
**Then** existiert dort eine `.editorconfig`-Datei

### Szenario 2: Standard-.NET-Konventionen sind konfiguriert

**Given** die `.editorconfig` existiert
**When** ich die Datei oeffne
**Then** enthaelt sie Regeln fuer:
  - Einrueckung (4 Spaces fuer C#)
  - Zeichensatz (UTF-8)
  - Zeilenenden
  - .NET-Namenskonventionen (PascalCase fuer public Members, etc.)
  - C#-spezifische Style-Regeln (var-Verwendung, Klammerstil, etc.)

### Szenario 3: Build laeuft weiterhin sauber

**Given** die `.editorconfig` wurde hinzugefuegt
**When** ich `dotnet build` ausfuehre
**Then** kompiliert die Solution erfolgreich ohne neue Fehler
