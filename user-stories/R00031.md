# User Story: R00031 - Health-Check HTTP-Endpoint

## Wer nutzt es?

Als **DevOps-Ingenieur oder Systemadministrator** moechte ich den Status eines von WindowsServicify verwalteten Dienstes ueber einen HTTP-Endpoint abfragen koennen, damit ich den Dienst in mein Monitoring-System (Prometheus, Grafana, Nagios, Uptime-Kuma) integrieren kann, ohne auf den Windows Service Manager oder Log-Dateien angewiesen zu sein.

## Mehrwert aus Nutzersicht

Bisher laesst sich der Prozess-Status nur ueber den Windows Service Control Manager (SCM) oder durch manuelles Lesen der Log-Dateien einsehen. Ein HTTP-basierter Health-Check-Endpoint ermoeglicht die automatisierte Ueberwachung durch gaengige Monitoring-Tools. Dies ist ein Alleinstellungsmerkmal gegenueber Wettbewerbern wie WinSW, Shawl und NSSM, die kein vergleichbares Feature bieten. Der Endpoint ist optional und beeintraechtigt bestehende Konfigurationen nicht.

## Akzeptanzkriterien

### Szenario 1: Konfiguration mit Health-Check-Port

**Given** der Benutzer fuehrt `--configure` aus
**When** nach dem Health-Check-Port gefragt wird
**And** der Benutzer einen gueltigen Port (z.B. 8080) eingibt
**Then** wird der Port in der `config.json` als `healthCheckPort` gespeichert
**And** die Konfiguration kann erfolgreich geladen werden

### Szenario 2: Konfiguration ohne Health-Check-Port (abwaertskompatibel)

**Given** der Benutzer fuehrt `--configure` aus
**When** nach dem Health-Check-Port gefragt wird
**And** der Benutzer die Eingabe leer laesst
**Then** wird `healthCheckPort` als `null` in der Konfiguration gespeichert
**And** kein HTTP-Endpoint wird gestartet

### Szenario 3: Bestehende config.json ohne HealthCheckPort

**Given** eine bestehende `config.json` ohne das Feld `healthCheckPort`
**When** die Konfiguration geladen wird
**Then** wird `HealthCheckPort` als `null` interpretiert
**And** der Service verhaelt sich wie bisher (kein HTTP-Endpoint)

### Szenario 4: Port-Validierung

**Given** eine ServiceConfiguration mit einem HealthCheckPort-Wert
**When** der Port ausserhalb des Bereichs 1024-65535 liegt
**Then** schlaegt die Validierung fehl mit einer entsprechenden Fehlermeldung

### Szenario 5: Health-Check bei laufendem Prozess

**Given** der Service laeuft und der Health-Check-Endpoint ist auf Port X konfiguriert
**And** der ueberwachte Prozess laeuft korrekt
**When** ein HTTP-GET-Request an `http://localhost:X/health` gesendet wird
**Then** antwortet der Endpoint mit HTTP 200
**And** der Response-Body ist JSON: `{"status":"healthy","process":"running","uptime":"HH:mm:ss"}`

### Szenario 6: Health-Check bei gestopptem Prozess

**Given** der Service laeuft und der Health-Check-Endpoint ist auf Port X konfiguriert
**And** der ueberwachte Prozess ist gestoppt oder gecrasht
**When** ein HTTP-GET-Request an `http://localhost:X/health` gesendet wird
**Then** antwortet der Endpoint mit HTTP 503
**And** der Response-Body ist JSON: `{"status":"unhealthy","process":"stopped"}`

### Szenario 7: Kein Endpoint ohne konfiguriertem Port

**Given** eine ServiceConfiguration mit `HealthCheckPort: null`
**When** der Service gestartet wird
**Then** wird kein HTTP-Listener gestartet
**And** kein Port wird belegt

### Szenario 8: Endpoint nur im Service- und Testrun-Modus

**Given** der Health-Check-Port ist konfiguriert
**When** der Dienst im Service-Modus oder Testrun-Modus laeuft
**Then** ist der `/health`-Endpoint erreichbar
**And** der Endpoint wird beim Stoppen des Dienstes sauber heruntergefahren

### Szenario 9: Monitoring-Kompatibilitaet

**Given** der Health-Check-Endpoint laeuft
**When** ein Monitoring-Tool (Blackbox-Exporter, Uptime-Kuma) den Endpoint abfragt
**Then** ist die Response-Time unter 100ms
**And** das JSON-Format ist strukturiert auswertbar
