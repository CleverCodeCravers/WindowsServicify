# R00015: .gitignore vervollstaendigen

## User Story

**Als** Entwickler im WindowsServicify-Projekt
**moechte ich** eine vollstaendige .NET-konforme .gitignore haben,
**damit** Build-Artefakte, IDE-spezifische Dateien und temporaere Dateien nicht versehentlich ins Repository gelangen und das Repository sauber bleibt.

## Mehrwert

Eine unvollstaendige .gitignore fuehrt dazu, dass Entwickler versehentlich generierte Dateien (bin/, obj/, .vs/) committen. Das blaaeht das Repository auf, verursacht Merge-Konflikte bei Build-Artefakten und erschwert Code-Reviews. Eine vollstaendige .gitignore verhindert diese Probleme praeventiv.

## Szenarien

### Szenario 1: Standard-.NET-Build-Artefakte werden ignoriert

**Given** die .gitignore wurde mit Standard-.NET-Patterns aktualisiert
**When** ein Entwickler `dotnet build` ausfuehrt und `git status` prueft
**Then** erscheinen `bin/` und `obj/`-Verzeichnisse nicht als ungetrackte Dateien

### Szenario 2: IDE-spezifische Dateien werden ignoriert

**Given** die .gitignore enthaelt Patterns fuer gaengige .NET-IDEs
**When** ein Entwickler mit Visual Studio, Rider oder VS Code arbeitet
**Then** werden IDE-spezifische Dateien wie `.vs/`, `*.user`, `*.suo`, `*.DotSettings.user` ignoriert

### Szenario 3: Bestehender .idea-Eintrag bleibt erhalten

**Given** die bisherige .gitignore enthielt `/Source/WindowsServicify/.idea`
**When** die .gitignore aktualisiert wird
**Then** ist der Eintrag `/Source/WindowsServicify/.idea` weiterhin vorhanden

### Szenario 4: Bereits getrackte Dateien bleiben unberuehrt

**Given** die neue .gitignore wurde eingefuegt
**When** geprueft wird ob getrackte Dateien von den neuen Patterns betroffen sind
**Then** gibt es keine getrackten Dateien die nun ignoriert wuerden (kein Datenverlust)

### Szenario 5: Testresultate und NuGet-Pakete werden ignoriert

**Given** die .gitignore enthaelt Patterns fuer `TestResults/` und `packages/`
**When** Tests ausgefuehrt oder NuGet-Pakete heruntergeladen werden
**Then** erscheinen diese Verzeichnisse nicht in `git status`
