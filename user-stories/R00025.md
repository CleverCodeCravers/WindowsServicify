# User Story: R00025 - CI-Workflow auf Release-Standard modernisieren

## Wer nutzt es?

Als **Entwickler und Release-Manager des WindowsServicify-Projekts** moechte ich einen CI-Workflow haben, der bei Push auf main automatisch Build und Tests ausfuehrt und bei manuellem Ausloesen (workflow_dispatch) zusaetzlich einen vollstaendigen Release-Zyklus mit Semver-Tagging und GitHub-Release durchfuehrt, damit Releases zuverlaessig, reproduzierbar und versioniert erstellt werden koennen.

## Mehrwert aus Nutzersicht

Der bisherige Workflow hat keinen automatischen Release-Mechanismus -- Releases erfordern manuelle Tag-Erstellung. Durch die Modernisierung auf das Zwei-Job-Muster (prepare + build-and-release) mit workflow_dispatch und automatischer Semver-Berechnung wird der Release-Prozess standardisiert. Die Build-Zeit-Versionierung stellt sicher, dass jedes Artefakt eindeutig einer Version zugeordnet werden kann. NuGet-Caching und Coverage-Reports bleiben erhalten.

## Akzeptanzkriterien

### Szenario 1: Push auf main fuehrt nur Build und Tests aus

**Given** ein Entwickler pusht Aenderungen auf den main-Branch
**When** der CI-Workflow ausgeloest wird
**Then** werden Build und Tests ausgefuehrt
**And** es wird KEIN Tag erstellt
**And** es wird KEIN Release erstellt
**And** Coverage-Report wird als Artefakt hochgeladen

### Szenario 2: workflow_dispatch loest vollstaendigen Release aus

**Given** ein Release-Manager loest den Workflow manuell aus
**When** er Major, Minor oder Patch als Bump-Typ auswaehlt
**Then** wird die naechste Semver-Version basierend auf dem hoechsten bestehenden Tag berechnet
**And** Build und Tests werden ausgefuehrt
**And** die Version wird zur Build-Zeit gesetzt (/p:Version=X.Y.Z)
**And** ein Semver-Tag (vX.Y.Z) wird erstellt und gepusht
**And** ein GitHub-Release mit dem ZIP-Artefakt wird erstellt

### Szenario 3: Semver-Berechnung interpretiert Tags korrekt

**Given** der hoechste bestehende Tag ist v1.8 (ohne Patch-Segment)
**When** die Semver-Berechnung ausgefuehrt wird
**Then** wird v1.8 als v1.8.0 interpretiert
**And** bei einem Patch-Bump ergibt sich v1.8.1
**And** bei einem Minor-Bump ergibt sich v1.9.0
**And** bei einem Major-Bump ergibt sich v2.0.0

### Szenario 4: NuGet-Caching bleibt erhalten

**Given** der CI-Workflow wird ausgefuehrt
**When** NuGet-Pakete wiederhergestellt werden
**Then** wird der NuGet-Cache basierend auf .csproj-Dateien verwendet
**And** bei Cache-Hit werden keine Pakete erneut heruntergeladen

### Szenario 5: Coverage-Report bleibt erhalten

**Given** der CI-Workflow wird ausgefuehrt
**When** Tests erfolgreich durchlaufen
**Then** wird die Code-Coverage via XPlat Code Coverage gesammelt
**And** der Coverage-Report wird als Artefakt hochgeladen

### Szenario 6: Build-Artefakt ist ein self-contained Single-File fuer Windows

**Given** der Build-Workflow wird ausgefuehrt
**When** die Publish-Phase erreicht wird
**Then** wird ein self-contained Single-File fuer win-x64 erstellt
**And** PDB-Dateien werden entfernt
**And** das Ergebnis wird als ZIP-Datei bereitgestellt

### Szenario 7: Zwei-Job-Muster mit korrekter Abhaengigkeit

**Given** der Workflow wird per workflow_dispatch ausgeloest
**When** der prepare-Job die Version berechnet hat
**Then** wird der build-and-release-Job erst danach ausgefuehrt
**And** er nutzt die berechnete Version als Input

### Szenario 8: Workflow hat korrekte Permissions

**Given** der Workflow wird ausgefuehrt
**When** ein Release erstellt werden soll
**Then** hat der Workflow die Permission contents:write
**And** Tags und Releases koennen erfolgreich erstellt werden
